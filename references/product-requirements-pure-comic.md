# Story Video Factory - Pure Comic First Requirements

Status: M0 draft for development review
Primary goal: define the first executable requirements for Claude Code development.

## 1. Product Positioning

Story Video Factory is a reviewed story-content production system for Douyin and similar platforms.

The first product priority is pure comic production. Comic-to-video and direct video are downstream extensions.

As of V1, `pure_comic` is the default project mainline. The canonical operating guide is `references/modes/pure-comic.md`; `comic_to_video` and `direct_video` remain later-phase extensions.

The system is not a one-click generator. It is a human-reviewed production line:

- The user approves story direction, outline, script, visual bible, final prompts when needed, and final selected images.
- The agent produces options, runs providers, organizes assets, performs QC, and prepares publishing material.
- Providers are selected by task: GPT-image for anchors and difficult key art, Jimeng for HD batch panels, MiniMax for later audio/video, composer for later video output.

## 2. Existing Skill Mapping

This document must not create a second truth. It extends the current repository structure.

Existing skills:

- `story-video-factory`: director skill; owns workflow, mode selection, review gates, provider routing, and handoff rules.
- `jimeng-browser-provider`: Jimeng browser automation; owns prompt submission, state tracking, HD download, resume, and manual intake support.
- `gpt-image-provider`: GPT-image anchors, character sheets, and difficult key panels in Codex environments.
- `douyin-publisher-pack`: publishing package. V1 must extend/reuse this for carousel output instead of creating a new publisher skill.
- `comic-video-composer`: later comic-to-video conversion. It expects `script.json` and `assets/images/<id>.<ext>`.
- `minimax-media-provider`: later TTS and video generation.

Mode guides and starter assets:

- `references/modes/pure-comic.md`: current default production path.
- `references/modes/comic-to-video.md`: later conversion path after a comic is approved.
- `references/modes/direct-video.md`: later video-provider path.
- `references/templates/pure-comic-episode/`: starter files for new pure comic episodes.
- `references/examples/star-in-uniform-001/`: text-only example extracted from the first M1 trial.

V1 additions should be references/templates under `story-video-factory` or root `references/`, not new skills, unless scripts or provider-specific automation appear.

## 3. V1 Product Goal

V1 should reliably produce a Douyin-ready pure comic carousel package.

Default output:

- 6-9 vertical comic panels.
- 9:16 aspect ratio.
- HD production images, not browser previews.
- Caption or narration text for each panel.
- Title, description, tags, and publishing notes.
- Traceable files so the comic can later become a video.

V1 success means:

> Given one story idea, the system can guide the user through review gates and produce a high-resolution, style-consistent, manually approved comic carousel with publishing material and a QC report.

## 4. V1 Workflow and Review Gates

V1 workflow:

1. Story idea intake.
2. Production mode selection: default `pure_comic`.
3. Outline draft and review.
4. Comic script / panel plan draft and review.
5. Visual bible draft and review. This includes style profile, character cards, and anchor/reference plan.
6. Prompt assembly and optional prompt review.
7. Image generation or manual image intake.
8. Candidate selection and rerun loop.
9. QC.
10. Publishing package output.

Hard gates that must stop for user approval:

- Outline before script.
- Comic script / panel plan before image generation.
- Visual bible before spending provider quota.
- Final selected images before publishing package or video conversion.

Soft gates that may continue by default unless the user asks to review:

- Style profile wording after visual bible approval.
- Provider-specific prompt wording.
- QC report generation.

## 5. Episode Directory Structure

Use one canonical structure that remains compatible with the existing composer.

```text
episodes/<episode-id>/
  episode-meta.json
  outline.md
  script.json
  storyboard.json
  prompts.json
  style-profile.md
  character-cards.md
  visual-bible.md
  quota-log.md
  prompt-rewrites.md
  assets/
    refs/
    images-raw/
    images/
    audio/
    video/
  output/
    publish/
    draft.mp4
    qc-report.md
```

Directory meaning:

- `assets/refs/`: style anchors, character sheets, and reference images. These are not necessarily production-resolution assets.
- `assets/images-raw/`: provider outputs and manually imported candidates.
- `assets/images/`: final selected production panels. Composer and later video tools read from here.
- `output/publish/`: publishing package generated by `douyin-publisher-pack`.

Do not introduce `selected-assets/` as a second final-image location in V1.

Panel IDs must be stable across files:

- Prompt id: `p001`, `p002`, ...
- Final image: `assets/images/p001.png`, `assets/images/p002.png`, ...
- Raw candidates: `assets/images-raw/p001_001.png`, `assets/images-raw/p001_002.png`, ...

## 6. Production Modes

The system should eventually support:

- `pure_comic`
- `comic_to_video`
- `direct_video`

V1 default:

- `pure_comic`

V1 still writes `script.json`, `storyboard.json`, and selected images in composer-compatible paths so Phase 3 can convert approved comics into videos.

## 7. Style System

The style system must be flexible. It must not restrict users to fixed presets.

Supported inputs:

- Preset style: a prepared example profile.
- Custom style: free user description.
- Hybrid style: user combines presets and custom ideas.

V1 should ship only 1-2 example profiles, not a full style library. A reusable style library is Phase 5.

The user can describe any style, such as cyberpunk, tomb-exploration adventure, old Hong Kong comics, folk horror, or a mixed visual direction. The system converts that free description into `style-profile.md` before prompt generation.

Required style profile fields:

- `style_name`
- `source_input`
- `visual_keywords`
- `color_palette`
- `line_and_texture`
- `camera_language`
- `composition_rules`
- `lighting_rules`
- `era_and_world_rules`
- `negative_style_rules`
- `provider_notes`
- `reference_assets`

## 8. Jimeng Style Discovery

Problem: Jimeng homepage examples often look better than hand-written prompts.

V1 must support a style discovery workflow:

1. The user may point to a Jimeng homepage/work-detail image or manually downloaded reference.
2. The agent records the reference under the episode's `assets/refs/`.
3. The agent extracts observable style traits: composition, color, lighting, line quality, subject treatment, texture, and prompt wording if visible on the work-detail page.
4. The agent translates those traits into `style-profile.md` and prompt assembly rules.
5. The final prompts should imitate the style traits, not copy protected characters or logos.

Manual style reference intake is allowed in V1. Automated scraping of Jimeng discovery pages is not required in V1, but the architecture should not block it. Global `references/style-samples/` is reserved for the Phase 5 style library, not V1.

## 9. Character Consistency

Character consistency is a first-class requirement, but V1 must be honest about what is implemented.

Consistency levels:

- Weak: repeat a fixed character card in every prompt.
- Anchor-based: create an approved character sheet or style anchor, then use it as a manual comparison baseline.
- Strong: use image-to-image, subject reference, or provider-specific reference workflows for every panel.

V1 default:

- Weak consistency plus approved anchor/reference baseline.

V1 does not claim strong automated consistency until image-to-image/reference workflows are implemented.

Required V1 behavior:

1. Create `character-cards.md`.
2. Optionally generate or import a character sheet / anchor image.
3. Ask the user to approve the visual bible.
4. Inject the character card into every panel prompt.
5. During QC, compare selected panels against the character card and anchor baseline.
6. If a panel fails, enter the rerun loop.

Character card fields:

- Name or role.
- Age impression.
- Face shape.
- Hair.
- Costume.
- Signature object or mark.
- Temperament.
- Common posture or gesture.
- Forbidden changes.

## 10. Prompt Assembly

Prompt assembly is the core of the pure comic pipeline.

The canonical rule lives in `references/prompt-assembly.md`.

Provider execution rule:

- V1 uses one provider-specific prompt file per runner, such as `prompts-jimeng.json` and `prompts-gpt-image.json`.
- The shared planning file may be `prompts.json`, but it must not be passed directly to a provider runner if it contains mixed providers.
- This avoids burning Jimeng credits on GPT-image-only entries before runner-side provider filtering exists.

Every provider prompt should be assembled from:

1. Panel intent.
2. Scene and action.
3. Character card block.
4. Style profile block.
5. Composition / camera block.
6. Era and object constraints.
7. Text policy.
8. Provider-specific suffix.

Negative rules should be rewritten into natural-language constraints for providers that do not support negative prompts directly.

Example: instead of only `negative_prompt: modern objects`, write `The scene contains no phones, cars, plastic items, neon signs, English letters, watermarks, or logos.`

## 11. Provider Strategy

Provider choice should be quality-first, not automation-first.

Default V1 routes:

- GPT-image: style anchors, character sheets, and difficult key panels.
- Jimeng: HD production panels and batch ideation.
- Manual intake: fallback for any automated provider failure.

Provider rules:

- Jimeng production images must come from `preview_download`, `downloads_dir`, or another explicitly production-approved method.
- Browser preview methods such as `preview_src` must not enter `assets/images/` unless the user explicitly marks the episode as low-resolution draft.
- GPT-image outputs may be accepted as reference assets even if they are below Jimeng production resolution.
- GPT-image outputs can become production panels only if the user accepts the resolution or an upscaling workflow exists.
- Every provider run must write state and metadata for reproducibility.

## 12. Asset Quality Gates

Asset classes:

- Reference asset: style anchor, character sheet, visual reference. Lower resolution is acceptable if not published directly.
- Draft candidate: exploratory generated image. May be low resolution.
- Production panel: selected final image for publishing or video conversion.

Production panel gate:

- Preferred minimum: 1440x2560.
- Ideal: original Jimeng PNG such as 3040x5404.
- Reject browser preview images such as 608x1080 unless `episode-meta.json` explicitly sets `draft_mode: true`.
- Accepted Jimeng methods: `preview_download`, `downloads_dir`, or configured production method.

QC must produce per-panel `pass`, `fail`, or `n.a.` results. See `references/qc-checklist.md`.

## 13. Rerun and Manual Intake Loop

The workflow is not linear. Failed panels must be rerunnable.

Rerun loop:

1. Panel fails QC or user rejects candidate.
2. Record reason in `prompt-rewrites.md` or provider state.
3. Revise only the necessary prompt blocks.
4. Regenerate that panel or import a manual candidate.
5. Re-run QC for that panel.
6. Copy approved candidate to `assets/images/<panel-id>.png`.

Manual intake:

- User may manually generate/download assets from Jimeng or another tool.
- Manual assets go to `assets/images-raw/` using stable candidate names.
- The same QC and selection rules apply.
- Manual intake is a first-class fallback, not an exception.

## 14. Quota and Budget

Each episode should track budget before generation.

`episode-meta.json` should include:

- `production_mode`
- `draft_mode`
- `target_panel_count`
- `provider_plan`
- `jimeng_budget`
- `gpt_image_budget`
- `minimax_budget`

`quota-log.md` should record:

- Provider.
- Prompt or task id.
- Number of attempts.
- Result path.
- Failure reason if any.
- Whether the asset was selected.

## 15. Publishing Package

V1 must reuse and extend `douyin-publisher-pack` for carousel output.

Pure comic publishing package includes:

- Final image order.
- Per-image caption or narration from `script.json`.
- Cover title.
- Douyin title options.
- Description.
- Tags.
- Comment prompt.
- Optional continuation hook.
- Risk notes.

Douyin carousel notes should be added to publisher references:

- Images must be readable on mobile.
- Cover should work as first image.
- Text should not rely on tiny in-image details.
- Do not assume video subtitles exist for carousel mode.

## 16. Open Decisions Resolved for V1

- Speech bubbles: not supported in V1. Use external captions/narration text.
- Selected assets: preserve provider original in `assets/images-raw/`; copy chosen production panels into `assets/images/` using stable panel IDs, preferably PNG for final workflow consistency.
- Low-resolution draft: allowed only when `draft_mode: true`; never silently treated as production.
- Presets: ship 1-2 example style profiles only. Full preset library is Phase 5.
- Style profiles: per-episode in V1. Global reusable style library comes later.
- Captions: `script.json` segment `text` is the single source of truth for carousel captions and future TTS narration. `storyboard.json` may contain draft caption notes, but accepted caption text must be promoted into `script.json`.

## 17. Future Plan

### Phase 1 - Pure Comic Pipeline

Goal: produce high-quality pure comic carousel packages.

Required capabilities:

- Story intake and outline review.
- Comic script / panel plan review.
- Style profile and character card templates.
- Prompt assembly schema.
- Jimeng HD image generation and manual intake.
- Candidate selection and rerun loop.
- QC report.
- Douyin carousel publishing package.

Acceptance criteria:

- One idea can produce a 6-9 image comic package.
- Each production panel passes the production asset gate or is explicitly marked draft.
- Each prompt uses the same style profile and relevant character card.
- QC report includes per-panel pass/fail/n.a. checklist.

### Phase 2 - Strong Character Consistency

Goal: make recurring characters more stable across panels.

Candidate capabilities:

- Jimeng image-to-image or reference-image workflow.
- GPT-image character sheet generation.
- Reference routing per provider.
- Character consistency scoring checklist.
- Manual rerun workflow improvements.

Acceptance criteria:

- A protagonist appears in at least 5 panels with recognizable clothing, face type, and temperament.
- Character reference assets are saved and reused.
- Failed consistency is visible in QC.

### Phase 3 - Comic-to-Video

Goal: convert approved pure comic packages into short videos.

Required capabilities:

- MiniMax TTS voice selection.
- Caption/subtitle generation.
- Ken Burns or camera-move composition.
- Optional MiniMax I2V for key panels.
- Video QC.

Acceptance criteria:

- A selected pure comic package can become a 9:16 video draft.
- The video uses approved images from `assets/images/`.
- Output has synced narration and readable subtitles.

### Phase 4 - Direct Video

Goal: generate short video shots directly through MiniMax or Jimeng when appropriate.

Direct video remains optional and should not block pure comic quality.

### Phase 5 - Style Library

Goal: save successful custom styles as reusable presets.

The style library can include Jimeng homepage-inspired samples, but it must store derived style traits and user-approved references rather than depending on unstable discovery-page scraping.

### Phase 6 - Publishing Operations

Goal: improve platform-ready publishing material, analytics notes, and reusable series conventions.

10. M3: QC report generator and carousel publishing pack.

Do not optimize direct video before the pure comic workflow is stable.

## 19. Known Risks

- Jimeng UI selectors can change. Keep manual intake as a supported fallback.
- Jimeng style can drift even with good prompts. Use style discovery, references, and human selection.
- Models may generate unwanted readable text. Treat this as a QC/rerun issue.
- Character consistency is not fully automated in V1.
- Provider quotas are limited. Use review gates and quota logs.
